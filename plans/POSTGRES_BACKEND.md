# Postgres Backend for Grist Documents

Store your Grist documents in PostgreSQL instead of SQLite. Your data
gets real Postgres types — booleans, dates, arrays — queryable by any
tool that speaks SQL. Grist keeps working exactly as before, but now
your data is also accessible from Python, Metabase, Grafana, or a
plain `psql` session.

## Quick start

```bash
# Start Postgres (if you don't have one)
docker run -d --name grist-pg \
  -e POSTGRES_USER=grist -e POSTGRES_PASSWORD=grist \
  -e POSTGRES_DB=gristdb -p 5432:5432 postgres:16

# Start Grist with Postgres backend
GRIST_DOC_BACKEND=postgres \
GRIST_DOC_POSTGRES_URL=postgres://grist:grist@localhost:5432/gristdb \
GRIST_SESSION_SECRET=your-secret \
node _build/stubs/app/server/server.js
```

Create a document in Grist as usual. Add some data. Then from another
terminal:

```bash
psql postgres://grist:grist@localhost:5432/gristdb

# See your documents as schemas
\dn doc_*

# Pick one and explore
SET search_path TO doc_your_document_id;
\dt

# Query with real types
SELECT "Name", "HireDate", "Active"
FROM "Employees"
WHERE "Active" = true
  AND "HireDate" > '2024-01-01';
```

Note the double quotes around table and column names. Grist uses
CamelCase names like `Table1` and `HireDate`, and Postgres lowercases
unquoted identifiers, so `SELECT * FROM Table1` looks for `table1`
and fails. Always quote: `SELECT * FROM "Table1"`. BI tools like
Metabase and Grafana handle this automatically — it's only an issue
in raw psql.

That `HireDate` is a real Postgres `date`. `Active` is a real
`boolean`. `SELECT *` gives you clean, typed data that any BI tool
can consume directly.

## What you get

Each Grist document becomes a Postgres schema. Each table in your
document becomes a Postgres table with native types:

| Grist column type | Postgres type |
|---|---|
| Text, Choice | `text` |
| Integer, Reference | `integer` |
| Numeric | `numeric` |
| Toggle (Bool) | `boolean` |
| Date | `date` |
| DateTime | `timestamptz` |
| Choice List | `text[]` |
| Reference List | `integer[]` |

If a cell has a value that doesn't fit its column type (like "banana"
in a numeric column, or a formula error), Grist stores it in a
companion `gristAlt_` column so it isn't lost. The main column shows
NULL for that cell. External tools can simply ignore the `gristAlt_*`
columns.

Formulas, undo, real-time collaboration — everything works the same
as with SQLite. The Python engine, the Grist UI, the API — nothing
changes from the user's perspective.

## What doesn't work yet

- **Snapshots / backups** — no pg_dump integration yet
- **SQL endpoint multi-statement safety** — the `pg` library's simple
  query protocol executes all statements in a string, unlike SQLite's
  node-sqlite3 which discards extras. The `select * from (...)` wrapper
  prevents injection within a single statement, but a semicolon can
  break out and add more. Needs the extended query protocol (named
  prepared statements) forced for untrusted SQL. Low priority since
  the endpoint already requires full document access.

## How it works

```
Grist UI / API
    ↓
ActiveDoc (same as before)
    ↓
PgDocStorage (new — replaces DocStorage for Postgres)
    ↓ native types + gristAlt_ companions
PgMinDB (new — translates SQLite SQL → Postgres SQL)
    ↓
PostgreSQL
```

The Python formula engine doesn't know or care about the backend.
It works with in-memory data and emits DocActions. PgDocStorage
translates those actions to Postgres DDL/DML with proper native
types.

`PgMinDB` handles the SQL dialect differences transparently:
- `?` parameters → `$1, $2, ...`
- `PRAGMA` → version table / no-op
- `BLOB` → `BYTEA`, `DATETIME` → `TIMESTAMPTZ`
- `INTEGER PRIMARY KEY` → `GENERATED BY DEFAULT AS IDENTITY`
- Auto-quoting of camelCase identifiers

Migrations are backend-agnostic via `MigrationUtils` helpers that
do the right thing for each backend (SQLite table rebuilds vs
Postgres `ALTER COLUMN`).

## Configuration

| Variable | Default | Description |
|---|---|---|
| `GRIST_DOC_BACKEND` | (unset = SQLite) | Set to `postgres` to enable |
| `GRIST_DOC_POSTGRES_URL` | `postgres://grist:grist@localhost:5432/gristdb` | Connection string |

The home database (user accounts, workspaces) is separate and
unchanged — it still uses whatever `TYPEORM_*` is configured.
The Postgres doc backend only affects document data storage.

Both backends can coexist on the same Postgres instance (different
databases or schemas). SQLite remains the default for anyone who
doesn't set these variables.

## Design decisions

### Why not just swap the storage layer?

The whole point is that Postgres tables should be **directly
queryable** by external tools with **native types**. A pure storage
swap (everything as BYTEA) would work but miss the value proposition.

### How non-conforming values work

Grist lets you put anything in any cell. Put "banana" in a numeric
column and it shows in red but is preserved. In Postgres with strict
types, that "banana" can't go in a `numeric` column.

Solution: each user data column gets a companion `gristAlt_<colName>`
column (bytea). When a value fits the type, it goes in the main
column and the alt is NULL. When it doesn't fit (errors, wrong-type
pastes, out-of-range values), the main column is NULL and the alt
holds the value in Grist's marshal binary format. This preserves
all Grist value types including Infinity, NaN, error tuples, and
nested lists.

```sql
-- External tools see clean data:
SELECT "Name", "Salary" FROM "Employees";
-- Name  | Salary
-- Alice  | 95000
-- Bob    | NULL    ← has an error, stored in gristAlt_Salary

-- Grist sees everything (merges main + alt on read)
```

Date values that aren't exact midnights (val % 86400 !== 0) go to
alt. DateTime values outside the year 0-9999 range go to alt.
Integer values outside the 32-bit range go to alt. This keeps the
native columns clean for external tools while preserving every
edge case for Grist.

### Schema per document

Each document gets its own Postgres schema (`doc_<id>`). This gives
clean namespace isolation, simple deletion (`DROP SCHEMA CASCADE`),
and natural access control boundaries. External tools do
`SET search_path TO doc_xxx` and then see clean table names.

### Shared initial SQL

The metadata table definitions (`_grist_DocInfo`, `_grist_Tables`,
etc.) come from the same `GRIST_DOC_SQL` source used by SQLite.
PgMinDB translates the SQL dialect differences. No separate schema
definition to maintain.

### Backend-agnostic migrations

Storage migrations use `MigrationUtils` helpers (`listTables`,
`listColumns`, `rebuildTable`) that do the right thing per backend.
SQLite uses the classic table-rebuild dance. Postgres uses `ALTER
COLUMN`. The migration code is shared.

## Remaining work

- **Snapshots / backup** — `pg_dump` per schema, integrate with S3
- **SQL endpoint multi-statement safety** — force extended query
  protocol for untrusted SQL (see "What doesn't work yet")
- **Filesystem-assuming tests** — a few tests check for `.grist`
  files on disk; need Postgres-aware assertions
